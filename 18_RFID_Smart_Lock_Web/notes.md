# Experiment 18 – ESP32 RFID Smart Lock Web Dashboard

## Overview

In this experiment, I built an RFID-based smart lock system using an ESP32, an RC522 RFID reader, and an SG90 servo motor.

The RC522 reads the UID of an RFID card or tag. The ESP32 compares the scanned UID to an authorized UID stored in the code. If the UID is authorized, the ESP32 unlocks the servo lock for a few seconds and then locks it again automatically.

The ESP32 also runs a local WiFi web server. A phone or computer browser connects to the ESP32 and displays a live dashboard with the lock state, last scanned UID, access result, servo angle, and scan counters.

This experiment combines hardware communication, access control logic, mechanical actuation, and web-based monitoring.

## Circuit Connections

### RC522 RFID Reader

The RC522 communicates with the ESP32 using SPI.

Connections:

RC522 SDA / SS → ESP32 GPIO21

RC522 SCK → ESP32 GPIO18

RC522 MOSI → ESP32 GPIO23

RC522 MISO → ESP32 GPIO19

RC522 RST → ESP32 GPIO22

RC522 IRQ → Not connected

RC522 GND → ESP32 GND

RC522 3.3V → ESP32 3.3V

Important:

The RC522 must be powered from 3.3V, not 5V.

---

### Servo Motor

The SG90 servo is powered from an external 5V supply.

Connections:

Servo Brown / Black → External 5V GND

Servo Red → External 5V +

Servo Orange / Yellow → ESP32 GPIO27

External 5V GND → ESP32 GND

Optional capacitor + → External 5V +

Optional capacitor - → External 5V GND

Important:

The ESP32 and external servo power supply must share the same GND. Without a common ground, the servo control signal has no proper reference.

Do not connect the external 5V supply to the ESP32 3.3V pin.

## Why the RC522 Uses 3.3V

The RC522 RFID reader module is designed to operate at 3.3V logic.

The ESP32 also uses 3.3V logic, so they are compatible.

The RC522 should not be connected to 5V. Supplying it with 5V can damage the module.

---

## SPI Communication

SPI stands for Serial Peripheral Interface.

It is a hardware communication protocol commonly used between microcontrollers and peripheral modules.

The RC522 communicates with the ESP32 using SPI.

Main SPI signals:

SCK:

Clock signal generated by the ESP32.

MOSI:

Data sent from the ESP32 to the RC522.

MISO:

Data sent from the RC522 to the ESP32.

SS / SDA:

Slave select pin. It tells the RC522 when the ESP32 is communicating with it.

RST:

Reset pin for the RC522 module.

IRQ:

Interrupt request pin. It was not used in this experiment.

---

## Why IRQ Was Not Connected

The IRQ pin can be used when the module needs to notify the ESP32 that an event occurred.

In this experiment, IRQ was not needed.

Instead, the ESP32 uses polling.

Polling means that the ESP32 repeatedly checks whether a new RFID card is present.

This is simpler and good enough for this project.

## RFID UID

Each RFID card or tag has a UID.

UID stands for Unique Identifier.

In the first test, the ESP32 printed the UID of the card to the Serial Monitor.

The authorized UID used in this experiment was:

24 84 01 07

In code form:

{ 0x24, 0x84, 0x01, 0x07 }

This UID was stored in the code as the authorized card.

## Access Control Logic

The ESP32 reads the scanned card UID and compares it to the authorized UID.

If all UID bytes match, access is granted.

If the UID size is different or at least one byte does not match, access is denied.

Basic logic:

Scanned UID == Authorized UID → Access Granted

Scanned UID != Authorized UID → Access Denied

This is a simple form of access control.

It is useful for learning, but it should not be considered secure enough for a real security system without stronger authentication.

---

## Servo Lock Behavior

The servo motor acts as a simple lock actuator.

In this experiment:

Locked angle = 10 degrees

Unlocked angle = 90 degrees

When an authorized card is scanned:

1. Access is granted
2. The servo moves to the unlocked angle
3. The lock remains open for 3 seconds
4. The servo returns to the locked angle

When an unauthorized card is scanned:

1. Access is denied
2. The servo remains locked

---

## External 5V Supply for Servo

The servo motor is powered from an external 5V supply.

This is important because servo motors can draw short current spikes when they start moving.

If the servo is powered directly from the ESP32 board, these current spikes may cause voltage drops and reset the ESP32.

Using an external 5V supply makes the system more stable.

The grounds must still be connected together:

External 5V GND → ESP32 GND

---

## Optional Capacitor

A 100uF capacitor can be connected across the servo supply.

Capacitor + → External 5V +

Capacitor - → External 5V GND

The capacitor helps reduce short voltage dips when the servo starts moving.

If using an electrolytic capacitor, polarity matters. The stripe usually marks the negative side.

## Web Dashboard

The ESP32 runs a local web server.

A phone or computer browser opens the ESP32 IP address.

The dashboard displays:

- Lock state
- Last scanned UID
- Access result
- Servo angle
- Total scan count
- Granted scan count
- Denied scan count
- JSON response

The dashboard updates automatically by sending repeated requests to the ESP32.

---

## HTTP Request

The browser periodically sends:

GET /data

This asks the ESP32 for the latest system state.

The ESP32 responds with JSON data.

---

## JSON Response

Example JSON response:

{
  "lock_state": "Unlocked",
  "last_uid": "24 84 01 07",
  "access_result": "Access Granted",
  "servo_angle": 90,
  "total_scans": 1,
  "granted_scans": 1,
  "denied_scans": 0,
  "status": "ok"
}

This JSON response contains both the RFID access result and the current actuator state.
